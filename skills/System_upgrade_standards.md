# 架构升级与迁移规范 (Golden Standard)

## 1. 迁移上下文与逻辑审计
- **核心任务**：将业务从 **[旧系统名]** 迁移至 **[新系统名]**。
- **先思后行**：在迁移具体功能前，必须审视老业务逻辑。若发现缺陷、冗余或可重构的点，**必须先列出建议清单**，与我交流确认后再实施重构，禁止盲目照搬老代码。

## 2. 编码规约 
- **黄金标准**：所有编码风格、包结构、命名习惯、注解使用必须**严格参考 `system` 模块**。
- **对象规范**：`VO`  / `DO` 对象层次必须齐全。禁止在不同层级间透传实体类，禁止省略转换过程。
- **持久层规范**：
    - 核心集成 **MyBatis-Plus**。
    - 基础 CRUD 及简单查询必须使用 MP 内置方法
    - 仅在涉及复杂统计、多表联查等 MP 无法胜任的场景时，才允许编写 `Mapper.xml`。
- **建表规范**：新生成的表结构必须遵循新系统规约，强制包含审计字段（如 `creator`, `create_time`, `updater`, `update_time`, `deleted` 等）。

## 3. 字典管理 
- **严禁硬编码**：禁止在前后端代码中硬编码任何状态值、类型值（如 `status == 1`）。
- **字典优先**：必须使用新系统完善的字典管理机制。在处理业务状态时，应引用对应的字典常量或调用字典服务。
- **未知对齐**：若老系统字段在新系统中缺少对应字典项，请在迁移报告中提出，待我确认字典 Code 后再编写。

## 4. 工具类与复用
- **禁止造轮子**：在编写逻辑前，必须先检索新系统前后端已有的工具类（后端参考 `system` 模块引用的 common 包或 utils 目录，前端参考src\components 目录）。
- **公共沉淀**：若需提取新的公共方法，必须按照 `system` 模块的分类逻辑，创建或扩展到对应的 `utils` 路径下。

## 5. 交互工作流 
1. **分析阶段**：读取老系统目标功能代码，并索引新系统 `system` 模块的实现模式。
2. **报告阶段**：在动笔前提交《迁移分析报告》，内容包括：
    - 老系统业务逻辑缺陷及重构建议。
    - 拟定新系统的 `VO` / `DO` 定义。
    - 涉及的字典项及需要调用的工具类清单。
3. **实施阶段**：在我回复“确认”或讨论达成一致后，开始分层编写代码。
