# 架构升级与迁移规范（参考资料）

## 目标
- 将业务从旧系统迁移到新系统
- 迁移过程中优先做业务审计与结构重构，而不是照搬旧代码

## 迁移前必须检查
- 老业务逻辑是否有缺陷、冗余、历史包袱
- 新系统 `system` 模块是否已有成熟实现模式
- 是否已有可直接复用的工具类、组件、字典、公共服务
- 状态字段、类型字段是否需要通过字典重新对齐

## 编码要求
- 所有编码风格、包结构、命名习惯、注解使用严格参考 `system` 模块
- `VO` / `DO` 对象层次必须齐全
- 禁止不同层之间直接透传实体类
- 基础 CRUD 与简单查询优先使用 `MyBatis-Plus`
- 复杂统计、多表联查等场景才允许写 `Mapper.xml`
- 新表必须包含审计字段：`creator`、`create_time`、`updater`、`update_time`、`deleted`

## 字典要求
- 禁止硬编码状态值、类型值
- 必须优先使用新系统字典机制
- 若老系统字段缺少对应字典项，应在迁移报告中提出，等待确认后再实现

## 复用要求
- 后端优先检查 `system` 模块依赖的 common 包或 utils 目录
- 前端优先检查 `src/components` 等公共组件目录
- 如需抽取公共方法，应按既有分类沉淀到对应 utils / 公共目录

## 标准工作流
1. 分析旧系统目标功能代码
2. 检索新系统 `system` 模块的实现模式
3. 输出《迁移分析报告》
4. 等用户确认
5. 分层实施迁移与重构
