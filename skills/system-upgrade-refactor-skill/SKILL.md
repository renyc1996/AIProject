# System Upgrade Refactor Skill

把“老系统升级到新系统”的经验沉淀成一个稳定工作流，避免机械搬运旧代码，优先做逻辑审计、模式对齐和分层迁移。

## 适用场景

当用户出现以下诉求时，优先使用本技能：
- 将旧系统业务迁移到新系统
- 参考新系统 `system` 模块重构旧模块
- 对老代码做升级改造、业务重写、分层治理
- 需要先出《迁移分析报告》，再等确认后实施
- 需要统一 VO / DO、字典、工具类复用、MyBatis-Plus 使用边界
- 需要把旧接口、旧表结构、旧状态流转迁移到新框架
- 需要评估“哪些照搬，哪些重构，哪些废弃”
- 用户表达比较口语化时，也要识别，例如：
  - “老项目要升级，帮我先捋一下怎么迁”
  - “这个旧模块别直接抄，按新系统规范重做”
  - “先帮我出方案，确认后再改代码”
  - “参照 system 模块，把这块业务重新落一遍”

## 不要误用的场景

以下情况通常不需要触发本技能：
- 只是修一个明确的小 bug，没有迁移、升级、重构背景
- 只是新增一个独立小功能，且与旧系统迁移无关
- 只是让你解释某段代码，而不是要迁移或改造
- 只是做纯前端样式调整，没有涉及新旧系统对齐

## 核心原则

1. 先思后行，不盲目照搬旧代码。
2. 先分析旧逻辑缺陷、冗余、可重构点，再给出建议清单。
3. 新实现严格参考新系统 `system` 模块的编码风格、包结构、命名习惯、注解方式。
4. 分层明确，`VO` / `DO` 必须齐全，禁止跨层透传实体。
5. 优先复用新系统已有工具类、组件、字典、公共能力。
6. 未确认的字典项、边界规则、业务取舍，先报告再实施。

## 工作流程

### 第一步：分析与对齐
先读取旧系统目标功能代码，同时检索新系统 `system` 模块中的对应实现模式。

分析时重点检查：
- 旧逻辑是否存在缺陷、冗余、历史兼容包袱
- 哪些逻辑应该保留，哪些更适合借机重构
- 新系统是否已有相同能力、公共工具、标准组件
- 数据结构、字段语义、状态流转是否需要重新建模

### 第二步：先交付《迁移分析报告》
在真正写代码前，先给用户一个简洁但完整的分析报告，至少包含：
- 旧系统业务逻辑缺陷与重构建议
- 拟定的新系统 `VO` / `DO` 设计
- 涉及的字典项清单
- 计划复用的工具类 / 公共方法 / 组件清单
- 是否需要新增公共工具，以及建议放置位置
- 哪些问题需要用户先确认

如果发现旧系统字段在新系统里缺少对应字典项，要明确指出并等待用户确认字典 Code。

### 第三步：等待确认后再实施
只有在用户明确回复“确认”或已经对方案达成一致后，才开始正式编码。

如果用户还没确认，就继续讨论方案，不要直接开改。

### 第四步：实施时遵守以下技术约束

#### 1. 编码规约
- 严格参考新系统 `system` 模块。
- 包结构、类命名、注解用法、返回风格保持一致。
- 不要引入和 `system` 模块风格冲突的新写法。

#### 2. 对象分层
- `VO`、`DO` 必须齐全。
- 不同层之间通过转换对象交互。
- 不要把数据库实体直接透传到接口层或前端层。

#### 3. 持久层规范
- 默认使用 `MyBatis-Plus` 处理基础 CRUD 和简单查询。
- 只有在复杂统计、多表联查、MP 明显不适合时，才编写 `Mapper.xml`。
- 如果写 `Mapper.xml`，要在说明里交代为什么不能直接用 MP。

#### 4. 建表规范
新增表结构必须遵循新系统规范，并包含审计字段，例如：
- `creator`
- `create_time`
- `updater`
- `update_time`
- `deleted`

如果项目还有租户、部门、组织等标准字段，也先检查 `system` 模块后对齐。

#### 5. 字典管理
- 严禁在前后端硬编码状态值、类型值、枚举数字。
- 必须优先使用新系统字典管理机制。
- 可以引用字典常量，或调用字典服务。
- 如果缺少字典项，先在报告里提出，不要擅自拍板。

#### 6. 工具类与复用
编码前，先检索新系统已有能力：
- 后端优先检查 `system` 模块依赖的 common 包、utils 目录
- 前端优先检查 `src/components` 等公共组件目录

如果确实需要沉淀新公共方法：
- 按照 `system` 模块现有分类方式落到对应 `utils` / 公共目录
- 不要随意创建风格不一致的新工具类

## 决策准则

遇到分歧时，优先遵循：
1. 新系统标准优先于旧系统历史实现
2. 清晰分层优先于省事透传
3. 字典与公共能力优先于硬编码和重复造轮子
4. 先沟通确认优先于擅自重构业务规则

## 例子

**高概率触发示例：**
- “这是老系统升级项目，帮我先分析一下这个模块怎么迁到新系统，参考 `system` 模块。”
- “把旧业务迁移到新框架，但别直接复制旧代码，先给我方案。”
- “这个遗留模块要重构，先审一下旧逻辑，再按新系统规范设计 `VO/DO`。”
- “你先输出迁移分析报告，确认后再开始改代码。”
- “旧表和旧接口都要升级，顺便把字典和状态流转梳理一下。”

**用户说：**
“这是一个老系统框架升级，业务重构的内容，先帮我看看怎么迁移到新系统，参考 system 模块来做。”

**你应该：**
1. 先读取旧系统相关代码与新系统 `system` 模块模式。
2. 先输出《迁移分析报告》。
3. 明确列出缺陷、建议、VO/DO、字典项、工具类复用点。
4. 等用户确认后再开始实现。

**用户说：**
“把老模块直接照着搬过来就行。”

**你应该：**
不要直接照搬。先解释为什么要先做逻辑审计，并给出最小必要的迁移分析与风险提示。

**用户说：**
“这个功能只是从 A 系统搬到 B 系统，你先直接写吧。”

**你应该：**
先提醒用户迁移前需要完成最小必要分析，并至少给出：
1. 旧逻辑是否有历史问题
2. 新系统 `system` 模块里可复用的模式
3. `VO` / `DO` 和字典设计草案
4. 待确认事项

**用户说：**
“参考 system 模块，把订单这块老业务重新实现一版，先出文档。”

**你应该：**
先按分析模板输出方案文档，再等待确认，不要直接进入编码。
